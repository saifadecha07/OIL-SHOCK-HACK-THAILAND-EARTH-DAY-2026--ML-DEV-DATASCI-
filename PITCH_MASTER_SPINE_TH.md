# 1. Problem

## ปัญหาของเราไม่ใช่แค่ "ราคาน้ำมันขึ้น"

ปัญหาจริงคือ เมื่อเกิด oil shock ระดับโลก  
ผลกระทบต่อระบบพลังงานไทยไม่ได้เกิดขึ้นทันที

มันเกิดแบบ `หน่วงเวลา`

และนี่คือจุดที่ผู้วางแผนมักพลาด

เพราะในโลกจริง:

- ราคาต้นน้ำเปลี่ยนก่อน
- แต่สัญญาณปลายน้ำในไทยค่อย ๆ ปรากฏทีหลัง
- พอเห็นชัด ก็มักช้าเกินไปสำหรับการเตรียมรับมือ

### ประโยคเปิดที่ควรใช้

"โจทย์ของเราไม่ใช่แค่การรู้ว่าราคาน้ำมันขึ้นหรือไม่ แต่คือการรู้ว่าหลังจาก shock เกิดขึ้นแล้ว ระบบพลังงานไทยจะเริ่มได้รับผลกระทบเมื่อไร"

---

# 2. Solution

## SHOCKWAVE คืออะไร

SHOCKWAVE คือ interactive early-warning dashboard  
ที่จำลองการส่งผ่านของ oil shock จากตลาดโลก  
ไปสู่ตัวชี้วัดพลังงานของไทย

ผู้ใช้กำหนด scenario ได้เอง แล้วระบบจะคำนวณผลกระทบล่วงหน้า

### สิ่งที่ระบบทำ

- รับ shock ของ Brent
- รับ forecast horizon
- คำนวณ baseline forecast
- คำนวณ shocked forecast
- หา critical month
- แสดงผลผ่านกราฟและตาราง

### ประโยคอธิบาย solution

"เราเปลี่ยน oil shock ระดับโลกให้กลายเป็นสัญญาณเตือนภัยที่อ่านออกได้ในบริบทไทย ผ่าน dashboard ที่ผู้ใช้สามารถตั้ง scenario และดูผลกระทบล่วงหน้าได้ทันที"

---

# 3. New Assumption

## ความใหม่ของเราอยู่ที่ hidden lag

เราไม่ได้เสนอแค่ว่า oil shock มีผลต่อไทย

เราเสนอว่า:

`hidden lag ระหว่าง global shock กับ local impact คือสัญญาณสำคัญในตัวมันเอง`

ถ้าเราจับ lag นี้ได้  
เราจะรู้หน้าต่างเวลาเสี่ยงก่อนผลกระทบจริงจะเกิดชัด

### ประโยคขายความใหม่

"ความใหม่ของเราคือเราไม่ได้มองแค่ระดับราคาหรือแนวโน้ม แต่เรามอง hidden lag เป็นสัญญาณที่สามารถ operationalize เป็น early warning dashboard ได้"

---

# 4. Why Track A

## ทำไมงานนี้ตรงกับ Planetary Signals Lab

เพราะงานนี้คือการเชื่อม:

- `planetary signal` จากตลาดพลังงานโลก
- ไปสู่ `local Thai energy stress`

โดยใช้:

- `EIA Open Data`
- `DOEB Open Data`
- `time-series analysis`
- `early warning dashboard`

### ประโยคที่ควรใช้

"งานนี้อยู่ใน Track A เพราะเราใช้ข้อมูลพลังงานระดับโลกเพื่ออนุมานสัญญาณผลกระทบที่กำลังจะส่งผ่านเข้าสู่ระบบพลังงานไทย"

---

# 5. UI Explanation

## หน้าจอนี้มีไว้ทำอะไร

หน้าจอ SHOCKWAVE ไม่ได้มีไว้โชว์กราฟเฉย ๆ

มันถูกออกแบบให้คนดูเข้าใจ flow ได้เร็วมาก:

`ตั้ง scenario -> รัน simulation -> อ่านผลกระทบ -> ตีความช่วงเวลาเสี่ยง`

## โครง UI ที่ควรพูด

### ฝั่งซ้าย

เป็นส่วน `ควบคุม scenario`

ผู้ใช้สามารถ:

- ปรับ shock percentage
- ปรับ forecast horizon
- กด run simulation

### ตรงกลางด้านบน

เป็นส่วน `context + summary`

เอาไว้บอกว่า:

- ตอนนี้เรากำลังจำลองเรื่องอะไร
- core signal คืออะไร
- ผลกระทบหนักสุดอยู่ตรงไหน
- critical month คือเดือนไหน

### กลางหน้า

เป็นส่วน `chart comparison`

เอาไว้ดูว่า:

- baseline กับ shocked path เริ่มแยกกันเมื่อไร
- import volume กับ diesel sales ตอบสนองต่างกันอย่างไร

### ด้านล่าง

เป็นส่วน `forecast table`

เอาไว้:

- ลงรายละเอียดรายเดือน
- ใช้ตอบคำถามกรรมการ
- ยืนยันตัวเลขจากกราฟ

# 6. วิธีอธิบาย UI บนเวที

## เวอร์ชันสั้น 20 วินาที

"หน้าจอนี้แบ่งเป็น 3 ส่วนหลัก คือ ฝั่งซ้ายสำหรับตั้ง scenario, ตรงกลางสำหรับดูผลสรุปและกราฟ, และด้านล่างสำหรับดูตัวเลขรายเดือน เพื่อให้ผู้ใช้เห็นได้เร็วว่า shock ที่กำหนดจะเริ่มส่งผลต่อระบบพลังงานไทยเมื่อไรและแรงแค่ไหน"

## เวอร์ชันเดโมจริง

"ผมจะเริ่มจากกำหนดขนาด shock และช่วงเวลาที่ต้องการ forecast ก่อน จากนั้นกด run ให้ระบบคำนวณ แล้วเราจะอ่าน summary cards ก่อนว่าจุดที่น่ากังวลที่สุดคืออะไร จากนั้นค่อยดูกราฟว่าผลกระทบเริ่มแยกจาก baseline เมื่อไร และปิดท้ายด้วยตารางรายเดือน"

# 7. Statistical Validity

## ถ้ากรรมการถามว่าโมเดลน่าเชื่อถือยังไง

สิ่งที่พูดได้จากของจริงในงาน:

- ตรวจ stationarity ด้วย `ADF`
- ทำ differencing เมื่อจำเป็น
- ใช้ `Granger causality`
- ใช้ `VAR`
- ใช้ `IRF`
- มี `time-based holdout backtest`
- วัด error ด้วย `MAE`, `RMSE`, `MAPE`

### ประโยคตอบแบบคม

"เราไม่ได้ใส่ shock แบบ arbitrary แต่ผ่านกระบวนการทาง time-series ครบตั้งแต่ stationarity, lag-aware relationship, multivariate forecasting ไปจนถึง holdout backtest บนช่วงท้ายของข้อมูล"
